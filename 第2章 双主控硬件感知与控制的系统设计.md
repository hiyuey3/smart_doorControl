第2章　双主控硬件感知与控制的系统设计
校园智慧实验室的使用环境对门禁系统的物理可靠性与网络实时性有较高要求。为了解决传统物联网门禁单一芯片在处理高吞吐量图像流与维持高实时性硬件中断时容易产生的资源抢占与死机问题，本系统在边缘感知与执行层设计了异构双主控的协同架构。

2.1 异构主控协同与底层硬件电路设计
2.1.1 STM32执行侧与电磁锁驱动电路设计
STM32微控制器作为系统的底层执行中枢，主要负责电磁锁的驱动、门磁状态的实时监测以及其他外设（包括按键输入、指纹模块与屏幕组件）的驱动。由于实验室常采用大功率磁力锁或电控机械锁，其动作时会产生较大的瞬态电流与反向电动势，为保护微控制器等弱电系统不被损坏，本系统设计采用光电耦合器隔离的继电器驱动电路。

STM32的GPIO引脚输出高低电平信号，首先点亮光耦内部的发光二极管，光敏三极管导通后再驱动后级的功率三极管，最终控制继电器的吸合与断开。该设计实现了弱电控制逻辑层与强电执行层的物理电气隔离，大幅提升了STM32在复杂电磁环境下的抗干扰能力。

2.1.2 ESP32-S3摄像头与网关模块外围电路设计
ESP32-S3作为感知与网络中枢，外挂了OV2640图像传感器。为保障视频快照采集的帧率与稳定性，系统充分利用了ESP32-S3内置的硬件JPEG编码加速器以及外部扩展的PSRAM（伪静态随机存储器）。其较高的运算能力确保了图像帧数据能够高效、无阻塞地呈现给终端用户。

2.2 异构主控异步通信协议设计
2.2.1 协议数据帧结构设计与实现
为实现ESP32-S3与STM32之间的高效协同与指令透传，系统设计了一套基于异步串口UART的轻量级自定义帧协议。波特率设定为115200 bps。数据帧结构包含严格的帧头（0xAA 0x55）、动作指令码、数据长度、数据体以及校验和，有效防止了总线干扰导致的误动作。

1. 自定义数据帧结构定义

系统定义的标准数据帧长度固定为6个字节，其内存排布与功能定义如下表所示：

字节偏移量	字段名称	长度 (Byte)	取值示例	字段功能描述
Byte 0	HEAD_1	1	0xAA	固定帧头1，用于数据流起始同步
Byte 1	HEAD_2	1	0x55	固定帧头2，双字节帧头防误判
Byte 2	CMD	1	0x01	业务指令码，如：0x01开锁
Byte 3	LEN	1	0x01	数据体长度，设定长为1字节
Byte 4	DATA	1	0x01	具体数据内容或动作参数
Byte 5	CHECKSUM	1	0x03	累加校验和，保障数据在强干扰下的完整性
2.2.2 ESP32下发命令到STM32

// UART TX
void send_hex_to_stm32(uint8_t cmd, uint8_t data) {
  uint8_t frame[6] = { 0xAA, 0x55, cmd, 0x01, data, (uint8_t)((cmd + 0x01 + data) & 0xFF) };
  SerialSTM.write(frame, 6);
  Serial.printf("UART TX: 0x%02X\n", cmd);
}
// MQTT to UART Router
void business_command_router(String json_msg) {
  JsonDocument doc;
  if (deserializeJson(doc, json_msg)) return;

  const char* cmd = doc["cmd"];
  if (!cmd) return;

  if (strcmp(cmd, "open_door") == 0) send_hex_to_stm32(0x01, 0x01);
  else if (strcmp(cmd, "alarm") == 0) send_hex_to_stm32(0x04, 0x01);
  else if (strcmp(cmd, "query_status") == 0) send_hex_to_stm32(0x05, 0x00);
  else if (strcmp(cmd, "reboot") == 0) ESP.restart();
}
云端中枢将管理员的操作指令序列化为JSON格式后，通过MQTT协议下发至设备专属的 /iot/device/{mac}/down 主题。ESP32-S3的 business_command_router() 函数解析该JSON载荷后，将其映射为对应的十六进制指令码，并调用 send_hex_to_stm32() 将6字节数据帧写入UART发送缓冲区，经由串口1（TX引脚4、RX引脚21）透传至STM32执行侧。

2.2.3 ESP32接收STM32回传命令结果

// UART RX
void business_uart_listen() {
  while (SerialSTM.available()) {
    static uint8_t state = 0, rx_cmd, rx_len, rx_data;
    uint8_t b = SerialSTM.read();
    switch (state) {
      case 0:
        if (b == 0xAA) state = 1;
        break;
      case 1:
        if (b == 0x55) state = 2;
        else state = 0;
        break;
      case 2:
        rx_cmd = b;
        state = 3;
        break;
      case 3:
        rx_len = b;
        state = 4;
        break;
      case 4:
        rx_data = b;
        state = 5;
        break;
      case 5:
        if (b == ((rx_cmd + rx_len + rx_data) & 0xFF)) {
          String payload = "{\"type\":\"hw_report\",\"mac\":\"" + WiFi.macAddress() + "\",\"cmd\":" + String(rx_cmd) + ",\"data\":" + String(rx_data) + "}";
          if (mqttClient.connected()) {
            mqttClient.publish(topic_pub.c_str(), payload.c_str());
            Serial.printf("UART RX: CMD=0x%02X\n", rx_cmd);
          }
        }
        state = 0;
        break;
    }
  }
}
上行链路的接收逻辑采用非阻塞式有限状态机实现。business_uart_listen() 在主循环中被持续调用，以字节为单位逐帧解析来自STM32的回传数据。状态机共6个状态：逐步校验0xAA帧头、0x55帧头，依次锁存指令码、数据长度、数据体，最终在状态5中对校验和进行比对验证。仅当校验通过时，才将硬件事件以 type: hw_report 的JSON格式封装，并通过MQTT上报至云端 /iot/device/{mac}/up 主题，完成双向透传的闭环。

2.3 边缘侧图像采集与准实时推流设计
实验室安防系统对环境监控的实时性与可靠性有较高要求。为了在有限的校园网带宽下实现平滑的视觉审计，本系统在ESP32-S3侧设计了基于硬件加速的图像采集与HTTP异步推流机制。

2.3.1 硬件级JPEG压缩与双缓冲机制设计
ESP32-S3内部集成的相机接口支持DVP并行传输，并配合硬件JPEG编码器，能够实现高效的图像处理。系统选用OV2640图像传感器，将其输出格式配置为 PIXFORMAT_JPEG。通过ESP32-S3内部的硬件编码器，原始的像素信号在DMA传输过程中即可实时转化为压缩的二进制流，相比软件编码方式可极大降低CPU的运算负载。

2.3.2 PSRAM双缓冲架构设计
为避免采集速度与网络上传速度不匹配导致的画面撕裂，系统将 fb_count 配置为2，即在外部扩展的PSRAM中开辟了两组独立缓冲区：当DMA引擎向第一个缓冲区填充新图像帧时，CPU可以同步读取第二个缓冲区中的数据进行网络发送。这种"乒乓缓冲"机制保证了监控快照的连贯性，实现了图像采集与网络传输的流水线化并行。

2.3.3 基于HTTP协议的快照轮询上传逻辑设计
针对校园网复杂的网络拓扑及NAT穿透需求，本系统未采用复杂的视频流协议，而是设计了轻量级的HTTP POST快照轮询机制。ESP32每隔1秒（UPLOAD_INTERVAL_MS = 1000）通过 HTTPClient 将最新一帧的JPEG二进制数据以 Content-Type: image/jpeg 格式 POST 至后端接口 /api/device/upload_snapshot。后端收到后将图像帧缓存于内存字典 device_frames[mac] 中，移动端通过轮询 /api/device/snapshot/{mac} 拉取最新帧，后端以 Base64 编码的 JSON 格式统一返回。

由于ESP32-S3同时承担MQTT维护与UART通信任务，过高的推流频率会导致系统阻塞。系统通过比较当前时间戳与 frame_upload_state.last_upload_time 实现非阻塞式节流，在保证准实时感官体验的前提下，确保了主循环的流畅度并降低了ESP32的发热量。

2.4 边缘设备固件生命周期与状态流转
边缘侧固件的整体生命周期可划分为启动初始化、在线稳态运行与故障自恢复三个阶段。

启动初始化阶段：固件上电后，依次执行 framework_network_init() 完成WiFi接入，若预存的SSID不可达则自动进入SmartConfig一键配网模式；随后完成NTP时间同步（阿里云NTP服务器），保证事件日志时间戳的准确性；最后初始化OV2640摄像头与STM32串口，并建立与MQTT Broker（mqtt.5i03.cn:1883）的长连接，订阅专属的下行指令主题 /iot/device/{mac}/down。

稳态运行阶段：主循环以非阻塞的方式并发调度四项任务：① mqttClient.loop() 维持心跳保活并处理下行指令；② business_uart_listen() 轮询UART接收缓冲区，处理STM32的异步上报；③ 图像帧节流上传，将最新快照推送至云端缓存；④ 每隔固定间隔向MQTT主题 /iot/device/{mac}/up 发布包含设备IP、时间戳与心跳计数的心跳包（type: heartbeat），云端据此实时更新设备的在线状态与末次心跳时间。

故障自恢复阶段：系统通过连续失败计数器 frame_upload_state.consecutive_failures 跟踪上传异常。当WiFi断线时，ESP32-S3在主循环中检测 WiFi.status() 并自动触发重连。MQTT断线时，由 mqttClient.connect() 的轮询机制自动恢复订阅关系。设备在MQTT连接建立时同步配置了Last Will Testament（遗嘱消息），一旦异常掉线，Broker将自动向 /iot/device/{mac}/status 发布离线通知，云端随即将该设备状态置为 offline，确保了设备感知状态的最终一致性。

第3章　云端中枢与安全鉴权设计
3.1 云端中枢架构与数据库设计
3.1.1 极简RESTful架构分层设计
云端系统基于Python语言与轻量级Web框架Flask开发，采用严格的资源化分层设计。如图所示，该架构自下而上分为三层：

数据持久层：基于Flask-SQLAlchemy构建的对象关系映射（ORM），通过混入模式抽象出通用基类，屏蔽底层SQL差异，同时通过 db.create_all() 与迁移函数 _ensure_device_columns() 实现数据库表结构的安全演进；
业务逻辑层：包含鉴权装饰器、权限拦截器、日志辅助类等横切关注点，与具体业务路由解耦；
资源请求调度层：通过Flask蓝图（Blueprint）将API路由（/api/*）与网页管理后台路由（/admin/*）独立注册，实现接口与视图的物理分离。
系统在 create_app() 工厂函数中完成以上三层的装配与初始化，确保了应用配置的集中管理与测试时的可替换性。

3.1.2 数据库解耦模型与实体关系设计
为应对实验室频繁的人员流动与设备更迭，系统采用关系型数据库并构建了高度抽象的ORM模型。

在数据库底层设计中，系统引入了混入模式（Mixin Pattern），在 core/models/mixins.py 中将所有资源实体共有的字段抽离为三个独立基类：

BaseIDMixin：统一提供自增整型主键 id，集中ID生成逻辑，避免各模型重复定义；
TimestampMixin：统一提供 created_at 与 updated_at 两个时间戳字段，updated_at 配置 onupdate 回调，实现自动维护，为全库审计与日志溯源提供标准化时间基准；
SerializerMixin：统一提供 to_dict() 序列化方法，通过SQLAlchemy的 inspect 机制自动遍历模型的全部列字段，并对 datetime 类型进行ISO 8601格式转换，消除了跨模型的序列化冗余代码。
核心业务逻辑依托于权限关联表 user_device_permissions。该表记录了用户标识符（user_id，外键关联 users.id）与设备物理地址（device_mac，外键关联 devices.mac_address）的多对多映射关系。不同于传统的静态布尔值授权，该表引入了枚举状态机，status 字段支持 pending、approved、rejected 三种状态的动态流转，能够精确实现实验室门禁权限的临时分配、人工审批与动态回收。此外，表中还设计了 apply_time（申请时间）、review_time（审批时间）、reviewed_by（审批人外键）字段，完整记录了权限从申请到审批的全链路生命周期。表级联合唯一约束 UNIQUE(user_id, device_mac) 从数据库层面保证了同一用户对同一设备不产生重复授权记录。

3.2 辅助工具类设计
3.2.1 统一数据流转与持久化管理
系统在 shared/db_helper.py 中封装了全局数据库辅助类 DatabaseHelper，对外以 db_helper 单例暴露。所有数据库增删改查操作均通过该类代理执行，每个方法均返回 (result, error) 二元组，强制调用方显式处理错误路径。该设计实现了数据操作的物理收敛，内置了自动事务回滚机制：在 add_and_commit()、update_and_commit()、delete_and_commit() 等方法中，一旦捕获到 SQLAlchemyError 异常，立即调用 db.session.rollback() 中止事务。此外，with_transaction 装饰器可应用于多表级联更新场景（如删除用户并级联清空其权限与日志记录），确保数据库状态的强一致性，从根源上杜绝了脏数据的产生。

3.2.2 标准化接口响应范式
移动端与云端的所有交互均遵循严格的RESTful规范。系统在 shared/response.py 中通过 ResponseHelper 类统一了网络请求的返回数据结构，无论业务执行成功与否，均返回包含 success 布尔标志、message 提示信息与 data 数据负载的标准化JSON格式：


// 成功响应
{ "success": true, "message": "成功", "data": { ... } }

// 错误响应
{ "success": false, "message": "无权限访问该设备", "error_code": "PERMISSION_DENIED" }
ResponseHelper 内置了 success()、error()、created()、paginated()、list_response() 以及 bad_request()、unauthorized()、forbidden()、not_found() 等一系列语义化快捷方法，实现了HTTP状态码与业务语义的统一映射，极大降低了前端解析多端异构数据的复杂度。

3.2.3 日志追踪与事件溯源体系
系统在 shared/logging.py 中设计了独立的日志辅助模块 LogHelper。每次访问事件（远程开锁、指纹识别、NFC刷卡等）发生时，系统自动从当前上下文提取时间戳与操作类型，并按 {unlock_method}_{timestamp}_{mac} 规则生成全局唯一的事件序列号 event_id，连同设备MAC地址、操作用户ID、开锁方式、快照URL一并写入 logs 表。LogHelper 针对不同识别方式提供了 create_remote_unlock_log()、create_fingerprint_log()、create_nfc_log() 等语义化快捷方法，实现了设备动作与用户操作的全链路可追溯。MQTT消息处理器在接收到硬件上报的通行事件（type: pass）时，同样通过 event_id 去重机制，避免同一事件被重复记录。

3.3 多模态身份认证授权防护设计
实验室门禁系统承载着高密级的物理安全诉求，其鉴权机制必须同时兼顾移动端交互便捷性与访问安全性。

3.3.1 移动端登录授权
系统支持密码核验及移动端免密两种登录模态，统一由 POST /api/login 接口处理，通过请求体中的 login_type 字段区分：password 模式执行用户名与SHA256密码哈希的比对；wechat 模式下，前端调用微信官方 wx.login() 接口获取一次性授权码（code），云端接收后携带 WX_APPID 与 WX_APPSECRET 向微信鉴权服务器发起请求，换取用户的唯一身份标识 openid。

鉴权通过后，云端不采用传统的基于服务端内存的Session会话机制，而是基于HMAC-SHA256算法（PyJWT 库），以应用的 SECRET_KEY 签发具备固定有效期（默认7天，可通过 JWT_EXPIRATION 环境变量配置）的无状态访问令牌（JSON Web Token）。令牌的Payload中隐式携带了 user_id、role 等核心身份声明。前端在后续的所有网络交互中，均通过 utils/request.js 封装的全局请求拦截器自动在 Authorization: Bearer <token> 请求头中注入该令牌，实现了高并发场景下的无状态安全鉴权。

云端的 token_required 装饰器（auth/decorators.py）负责验证：从请求头提取令牌→校验Bearer格式→用 jwt.decode() 验证签名与有效期→从Payload中取出 user_id 查询数据库→将用户对象注入Flask请求上下文 g.current_user，供后续路由函数直接调用。

3.3.2 安全拦截机制
在常规的物联网接口架构中，常存在失效对象级授权漏洞（BOLA/IDOR）：非法攻击者可通过抓包截获开锁接口，并恶意篡改URL中的设备MAC地址参数，从而控制其未获得真实授权的实验室门锁。

为封堵该越权漏洞，系统在 auth/permissions.py 中部署了 PermissionHelper.check_device_access() 全局权限拦截函数。当系统接收到下发远程开门指令的请求时，必须先穿透此拦截器。拦截器提取 g.current_user 中的用户身份凭证，并以 user_id 与目标 device_mac 为条件，反向查询 user_device_permissions 表中 status = 'approved' 的记录，以严格的访问控制列表（ACL）校验机制确认请求发起者是否具备该物理设备的访问权限。若无匹配记录，则直接返回HTTP 403状态码，从而实现了架构级别的水平越权防御。管理员角色（role = 'admin'）经特殊处理，默认拥有全部设备的访问权限，无需逐设备授权。

3.4 动态权限管理
实验室的设备归属与人员进出具备高度的动态性。系统在业务中枢引入了动态权限状态机机制，摒弃了静态的单一授权模型。

数据库的权限状态被严格设定为三种核心枚举值：pending（待审批）、approved（已批准）与 rejected（已拒绝）。

多向申请流转：系统管理员可通过网页管理后台 POST /admin/permissions/add 接口，主动为特定科研组成员批量分配实验室准入权限，权限状态直接初始化为 approved；同时，学生或科研人员也可在移动端主动发起门禁访问申请，此时权限状态机自动初始化为 pending，等待管理员审批。

可信审批闭环：管理员在网页管理后台的"权限审批"页面（/admin/permissions）对申请执行审批操作。通过 POST /admin/permissions/<id>?action=approve 或 action=reject 完成状态流转。一旦状态流转至 approved，用户即可在其移动端控制台实时查看到该门禁设备，完成物理权限的数字下发闭环；若科研项目结题或人员调离，管理员可通过 action=revoke 一键撤销权限，系统将从数据库中删除该授权记录，实时阻断后续的指令下发。业务逻辑层对每次状态变更均内置了合法性前置校验（approve/reject 仅可处理 pending 状态，revoke 仅可处理 approved 状态）并配合完整的异常捕获与事务回滚，确保状态机流转的原子性与一致性。

第4章　移动端交互设计
移动端应用作为智慧校园实验室门禁系统与终端用户直接交互的媒介，其界面合理性与交互流畅度直接决定了系统的工程可用性。本章详细论述了无状态令牌的无感刷新机制与准实时视频渲染引擎的客户端实现逻辑。

4.1 移动端应用界面设计
为了在有限的移动端屏幕内实现复杂安防信息的高效传递，本系统前端摒弃了冗余的视觉装饰，确立了以品牌蓝为主色调的工业极简设计语言。

视图层全面采用卡片化容器布局。全局页面背景统一设定为极浅灰（#f7f8fa，与 app.json 全局窗口配置一致），以提供舒适的视觉呼吸感；所有核心安防数据与操作模块均封装于背景为纯白、圆角设定为28响应式像素（rpx）且带有极微弱环境阴影的独立卡片组件内，在视觉上实现了功能模块的隔离与解耦。

在图标与引导元素的设计上，采用CSS伪元素技术与纯代码绘制状态引导图标，规避了外部图标库的引入，确保了应用界面的严谨性与跨平台一致性。底部TabBar包含"控制台"与"个人配置"两个核心入口，图标分别提供默认态与选中态（品牌蓝高亮），强化了用户的导航感知。

4.2 鉴权令牌的本地化管理与无感刷新机制
用户完成多模态登录后，云端签发的JWT访问令牌被持久化存储于客户端的本地缓存（wx.setStorageSync('token', token)）中。应用启动时（App.onLaunch），系统自动尝试读取本地缓存中的Token与用户信息，无需用户重复登录，实现了会话的自动恢复。

为解决原生 wx.request 接口产生的大量鉴权冗余代码问题，系统在 utils/request.js 中封装了全局网络请求拦截器。该拦截器具备两大核心能力：

其一，自动注入鉴权凭证：拦截器在发起每一个出站请求时，自动从本地缓存读取Token，并将其以 Authorization: Bearer <token> 格式注入请求头，业务页面代码无需关心Token的携带细节。

其二，全局401状态码监听：当拦截器捕获到云端返回的HTTP 401状态码时，系统判定当前令牌已过期或被权限拦截器阻断。随后自动执行 wx.removeStorageSync('token') 与 wx.removeStorageSync('user') 清空本地凭证，并以平滑的标准过渡动画将用户重定向至登录视图（/pages/login/index），有效阻断了僵尸越权会话的存留。

4.3 分页设计
登录页（pages/login/index）：支持微信一键授权与用户名/密码两种登录方式。登录成功后将Token与用户信息写入本地缓存，并跳转至控制台主页。

控制台页（pages/console/index）：门禁控制主视图。页面加载时调用 GET /api/user/devices 接口，该接口在云端权限表中筛选当前用户状态为 approved 的设备记录后返回。前端接收设备阵列后，通过数据绑定机制驱动视图层异步更新。针对硬件设备的网络状态，前端设计了映射字典，将底层 status 字段转化为胶囊状"药丸标签"：在线设备渲染为浅绿底深绿字标签；离线设备渲染为浅灰底深灰字标签。主操作按钮采用无阴影的纯实色品牌蓝填充，确保了用户在紧急开门场景下的极致操作确定性。

设备详情页（pages/device-detail/index）：沉浸式监控与多维控制视图。页面 onLoad 时同步启动三个并行任务：从全局数据中加载设备基本信息、调用 loadDeviceSnapshot() 获取首帧快照，以及通过 setInterval 启动快照定时刷新（轮询机制）与设备状态轮询。页面销毁（onUnload）时自动清理全部定时器，防止内存泄漏。

日志页（pages/logs/index）：展示通行事件列表，支持下拉刷新，记录包含设备名、识别方式（远程/指纹/NFC）与时间戳的完整事件流。

用户管理页（pages/users/manage）：管理员专属页面，支持查看用户列表、添加用户及为用户绑定指纹ID或NFC UID等操作。

个人配置页（pages/settings/config）：提供用户信息展示、API环境切换（开发/生产）、微信账号绑定/解绑等个性化配置入口。

4.4 网页管理后台界面设计
网页管理后台（/admin/*）基于Jinja2模板引擎渲染，采用Bootstrap框架构建响应式布局，遵循与移动端一致的极简设计语言，面向管理员提供完整的CRUD管理能力。

后台包含以下核心功能模块：

仪表盘（/admin/dashboard）：展示系统整体运行状态，包括在线设备数量、注册用户总数、待审批权限申请数量及近期通行事件统计，为管理员提供全局视图。

用户管理（/admin/users）：以表格形式列出所有注册用户，支持通过模态框添加新用户（填写学号/工号、姓名、角色）、编辑用户信息（修改姓名、角色、指纹ID、NFC UID），以及通过POST表单提交删除用户。编辑操作通过 POST /admin/users/<id>?action=edit 处理，删除通过 action=delete 区分，操作前均弹出确认对话框防止误操作。

设备管理（/admin/devices）：展示所有已注册门禁设备的MAC地址、名称、位置、状态与最后心跳时间，支持添加设备（MAC地址作为唯一标识）、编辑设备信息（名称、位置、房间号，MAC不可修改）与删除设备。路由结构与用户管理保持一致的RESTful风格。

权限审批（/admin/permissions）：提供全状态的权限管理能力。顶部展示待审批数量徽章，支持按 pending/approved/rejected 状态筛选。操作列根据当前记录状态动态渲染不同按钮：待审批记录显示"批准/拒绝"，已批准记录显示"撤销"。此外，"添加权限"功能允许管理员主动通过模态框选择用户与设备，直接创建已批准状态的权限记录。

通行日志（/admin/logs）：按时间倒序展示所有设备的通行事件记录，包含事件ID、触发设备、识别方式与发生时间，为安全审计提供完整的操作留痕。

后台所有数据修改操作均采用POST方法提交（HTML表单原生限制），通过 _method 隐藏字段或 action 参数区分具体操作类型，并内置了完整的异常捕获与数据库回滚机制，操作失败时在页面顶部以红色Alert形式展示具体错误原因，保证了管理界面的操作可靠性。

第5章　系统联调与功能实测
在完成边缘端硬件驱动、云端业务中枢及移动端交互系统的开发后，本章对系统进行了全链路的联合调试与功能验证。通过模拟真实校园实验室环境，对多模态登录、远程指令下发、准实时快照渲染等核心业务进行了压力测试，并详细记录了针对底层物理冲突与协议兼容性的调试过程。

5.1 联调环境搭建与参数配置
系统测试环境由物理实验室节点（边缘层）、华为云服务器（云端层）及多部移动终端（交互层）三层组成。云端部署了基于Docker Compose编排的Flask后端服务与Nginx反向代理，引入 werkzeug.middleware.proxy_fix.ProxyFix 中间件，确保 Flask redirect() 在反向代理后能够正确识别真实域名与协议，解决了登录成功后重定向至内网地址的问题。数据库采用SQLite，通过混入模式（Mixin）预设了多组测试账户与设备权限映射表，并预置了 admin/admin123 默认管理员账号（仅首次启动时自动创建）。ESP32-S3固件编译后烧录至实验室门禁节点，通过SmartConfig完成入网配置，连接至与云端服务器同一MQTT Broker（mqtt.5i03.cn:1883）。

5.2 移动小程序端设计验证
经实测，新用户首次打开微信小程序后，系统能够正确调用微信官方接口并下发具备BOLA防护能力的无状态JWT访问令牌。在控制台界面，系统仅向用户渲染其在数据库中状态为 approved 的实验室门禁节点。测试人员尝试手动修改网络请求中的设备MAC地址参数，系统权限拦截器均能实时识别非法越权动作并返回HTTP 403错误，验证了权限管理逻辑的鲁棒性。

5.2.1 异构双主控控制流联调
测试人员在移动端点击"远程开锁"按钮，云端中枢迅速将指令序列化为 {"cmd": "open_door"} 格式的MQTT载荷，通过 publish_command() 函数发布至 /iot/device/{mac}/down 主题。ESP32-S3接收到载荷后，经由 business_command_router() 非阻塞状态机解析，调用 send_hex_to_stm32(0x01, 0x01) 将6字节数据帧（0xAA 0x55 0x01 0x01 0x01 0x03）推入UART发送缓冲区。经串口波形分析仪观测，底层STM32执行主控能够在接收到帧头 0xAA 0x55 后的50毫秒内准确驱动隔离继电器动作，实现了端到端控制流的低延迟闭环。STM32完成执行后，将结果帧经UART回传至ESP32-S3，后者校验通过后将 type: hw_report 事件上报至MQTT，云端日志系统同步记录本次远程开锁事件，完整验证了"移动端→云端MQTT→ESP32→UART→STM32→继电器→MQTT上报→云端日志"的全链路数据流转。

5.2.2 准实时快照渲染验证
测试过程中对快照渲染的稳定性进行了专项验证。ESP32-S3以1秒为间隔将OV2640采集的JPEG帧POST至后端 /api/device/upload_snapshot 接口，后端以 normalize_mac() 函数统一MAC地址格式（支持带冒号、不带冒号、带连字符等多种输入格式统一转化为 AA:BB:CC:DD:EE:FF 标准格式）后存入内存缓存。移动端设备详情页轮询 /api/device/snapshot/{mac} 接口，后端将缓存中的JPEG二进制数据以 base64.b64encode().decode('utf-8') 编码后封装于JSON中返回。前端额外执行 .replace(/[\r\n]/g, "") 清理潜在换行符，再以 data:image/jpeg;base64,{data} 格式赋值给 <image> 组件的 src 属性，成功解决了微信小程序对Base64字符串的严格格式校验问题，快照渲染成功率达到100%。
